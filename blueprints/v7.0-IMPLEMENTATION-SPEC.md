# Vibe Science v7.0 TRACE — Implementation Spec

**Codename:** TRACE
**Status:** SPEC FOR IMPLEMENTATION
**Data:** 24 marzo 2026
**Baseline:** `vibe-science/` stato reale al 24/03/2026
**Scopo:** definire una specifica implementativa vera, dettagliata e atomica per l'evoluzione di Vibe Science da v6.0 NEXUS a v7.0 TRACE, senza deragliare dall'identita' del progetto

**Perche' TRACE:** v7 rende ogni decisione del sistema tracciabile — claim lifecycle nel DB, citazioni tracciate fino alla fonte, review con metriche persistite, benchmark artifact/recording pronti per confronti onesti, strict mode che rende visibile quando l'infrastruttura degrada. Se v6 NEXUS ha costruito il corpo, v7 TRACE gli da' la memoria verificabile di tutto quello che fa.

---

## 0. Come leggere questa spec

Questa non e' una wishlist.

Questa non e' una roadmap marketing.

Questa non e' una raccolta di idee "cool".

Questa e' una **spec di implementazione** che:

- parte dallo stato di fatto del plugin e della skill
- incorpora le analisi profonde gia' fatte
- traduce i gap verificati in work package atomici
- definisce dipendenze, confini, criteri di accettazione e ordine corretto

Questa spec deve permettere a un team di sub-agent di implementare v7 senza interpretazioni creative, senza scope creep e senza perdere l'identita' del progetto.

   L'identita' resta quella di integrity-first research runtime.

7. **No feature theatre.**  
   Ogni capability nuova deve essere misurabile o verificabile.

## 6. Principi architetturali di v7

### 6.1 Structured before heuristic

Il plugin non deve affidarsi solo a regex sulla prosa libera se possiamo avere artefatti strutturati.

Regola v7:

- se esiste uno schema, si usa quello
- se esiste un artefatto strutturato, quello ha priorita'
- il parsing euristico del markdown resta solo come compatibilita' transitoria

### 6.2 Human artifact + machine artifact, non uno contro l'altro

`CLAIM-LEDGER.md`, `SERENDIPITY.md`, review reports e `STATE.md` restano artefatti umani importanti.

Ma v7 introduce il principio:

- artefatto umano per leggibilita'
- struttura macchina per ingestion affidabile

La forma puo' essere una di queste due, in ordine di preferenza:

1. blocco strutturato embedded nel markdown
2. sidecar JSON/YAML opzionale dove serve davvero

Decisione v7:

per il core di v7 il path canonico non e' il sidecar esterno.

E' il **blocco strutturato embedded** dentro gli artefatti umani gia' esistenti, perche' e' il modo piu' aderente allo stato reale della skill e al workflow corrente del progetto.

**Contratto canonico del blocco embedded**

Per evitare divergenze tra work package, v7 usa un solo pattern sintattico condiviso:

- fenced markdown block tipizzato
- info string canonica:
  - ````vibe-claim`
  - ````vibe-seed`
  - ````vibe-review`
- payload interno in YAML semplice

Esempi canonici:

```text
```vibe-claim
id: C-003
event_type: CREATED
confidence: 0.72
confounder_status: PENDING
```
```

```text
```vibe-seed
seed_id: S-014
source: SALVAGED_FROM_R2
source_claim_id: C-003
score: 13
```
```

```text
```vibe-review
review_id: R2-009
review_mode: BATCH
claims_reviewed:
  - C-003
j0_score: 4
```
```

Regola:

- claim, seed e review non possono inventare tre sintassi diverse
- un solo parser di famiglia deve poter riconoscere tutti e tre i blocchi
- il sidecar resta opzionale e secondario
- il parser condiviso canonico vive in `plugin/lib/structured-block-parser.js`

### 6.3 Shared hotspot files con ownership stretta

Ci sono file hotspot che non vanno editati in parallelo da piu' worker senza coordinamento:

- `plugin/scripts/post-tool-use.js`
- `plugin/lib/gate-engine.js`
- `plugin/db/schema.sql`
- `evals/eval-runner.mjs`

Per questi file serve sempre un integrator owner.

### 6.4 Benchmark-first, non benchmark-later

Ogni capability v7 deve essere introdotta con:

- baseline
- test mirati
- criterio di miglioramento

### 6.5 One strict flag first

v7 non introduce subito tre profili completi `assistive/audit/strict`.

Primo passo:

- un flag unico di strictness
- visibilita' esplicita di `INTEGRITY_DEGRADED`
- fallimenti infrastrutturali resi rumorosi, non invisibili

### 6.6 Test proportionality

v7 non puo' aumentare la superficie del sistema senza aumentare la superficie di verifica.

Regola generale:

- ogni nuovo helper module deve portare test unitari dedicati
- ogni modifica a un hotspot file deve portare almeno un test di integrazione o regressione sul path modificato
- le migrazioni devono avere test di upgrade reale da stato v6

Il numero esatto finale di test non e' la metrica principale.

La metrica principale e' questa:

- nessun WP atterra senza nuova evidenza di verifica proporzionata al rischio introdotto

### 6.7 Merge discipline and rollback

v7 non usa la logica "mergiamo e poi sistemiamo".

Regola:

- se un WP rompe i test esistenti, quel WP non viene mergiato
- il sub-agent corregge la regressione prima del merge
- se la regressione non e' correggibile senza riscrivere il WP, il WP viene riprogettato
- non si accetta fix-forward lasciando test rotti nella mainline

### 6.8 UX and latency budget

v7 aumenta la potenza del runtime, ma non puo' trasformare ogni hook in un collo di bottiglia ingestibile.

Regole:

- `post-tool-use.js` ha un budget totale target di `<= 6000ms` anche nel caso peggiore v7
- la verifica citazioni e' l'unica componente autorizzata a consumare budget di rete sincrono significativo
- per singolo evento hook si tentano al massimo `3` verifiche citazionali sync
- il budget massimo sync dedicato alla rete resta `5000ms` per evento; oltre quel limite le citazioni restanti passano a `PENDING`
- i messaggi di blocco devono spiegare chiaramente all'utente cosa fare dopo, non solo che il gate ha fallito

Mitigazioni operative richieste:

- strict mode deve avere una procedura documentata di disattivazione
- lo stop blocking deve avere una procedura di emergenza esplicita, rumorosa e auditabile; non e' ammesso un bypass silenzioso

### 6.9 Versioning discipline

Il version bump non puo' falsare benchmark o report intermedi.

Regola:

- i benchmark baseline restano registrati come linea `6.x`
- i benchmark candidati pre-release usano etichette prerelease esplicite, ad esempio `7.0.0-rcN`
- il bump stabile di `package.json` a `7.0.0` avviene solo dopo che i criteri globali di v7 e il readiness gate sono verdi

Regola trasversale:

### WP-00 — Executable Eval Baseline

**Goal**  
Sostituire il mismatch "eval dichiarati vs YAML linter" con una scala di valutazione eseguibile e onesta per v7.0.

**Perche' esiste**  
Oggi `evals/eval-runner.mjs` valida soprattutto schema/artifact. Per v7.0 il release core deve dirlo esplicitamente e collegarlo ai regression hook test reali, senza fingere una behavioral automation completa che ancora non esiste.

**Output richiesti**

1. Rinominare esplicitamente il runner attuale come livello `schema`.
2. Introdurre una ladder a tre livelli:
   - `L0-schema`: validazione dei case YAML
   - `L1-hook-runtime`: regressioni sintetiche / hook-level su stdin/stdout e side effect reali
   - `L2-agent-behavior`: eval semi-end-to-end completi, esplicitamente fuori dal release core v7.0 se non implementati
3. Formalizzare `GC01-GC12` come suite classificata per livello di esecuzione, non solo documentale.
4. Creare `evals/BASELINE_V6.md` con numeri reali.
5. Definire un catalogo canonico di eventi sintetici per `L1-hook-behavior`, con payload JSON, exit code atteso e side effect atteso su DB/log.

**Ordine interno obbligatorio**

1. definire il catalogo degli eventi sintetici
2. costruire il catalogo / harness `L1-hook-runtime`
3. classificare i case esistenti per livello di esecuzione
4. eseguire la baseline e materializzarla in `evals/BASELINE_V6.md`

**File owner**

- `evals/eval-runner.mjs`
- `evals/README.md`
- nuova cartella `evals/harness/`
- eventuale integrazione con `plugin/lib/benchmark-reporter.js`

**Scelte progettuali**

- non promettere full Claude Code automation se non c'e'
- separare chiaramente "spec case" da "executed case"
- ogni case deve dichiarare il livello minimo a cui e' valido
- `L1-hook-runtime` non usa prompt per agenti: usa eventi sintetici che rispettano il protocollo stdin/stdout reale dei hook

**Acceptance criteria**

- il runner non puo' piu' chiamarsi behavioral se fa solo schema validation
- esiste una baseline documentata e registrabile per i case realmente eseguiti in v7.0
- i risultati possono essere registrati in `benchmark_runs`
- esiste una mini-libreria di eventi sintetici canonici per almeno i path critici di `PreToolUse`, `PostToolUse`, `SubagentStop`, `Stop`

**Out of scope**

- benchmark esterni completamente automatizzati

### WP-00A — Schema Migration Foundation

**Goal**  
Rendere evolvibile il DB reale di Vibe Science senza affidarsi solo a `CREATE TABLE IF NOT EXISTS`.

**Perche' esiste**  
v7 introduce nuove colonne e nuove tabelle. Su installazioni gia' esistenti di v6, rieseguire solo `schema.sql` non basta.

**Output richiesti**

1. tabella minimale `meta` o equivalente con `schema_version`
2. migration runner minimale in `plugin/scripts/setup.js`
3. supporto almeno per:
   - nuove colonne su tabelle esistenti
   - nuove tabelle v7
4. strategia idempotente con `try/catch` o check preliminari

**Decisione di spec**

v7 non introduce subito un framework di migration pesante.

Introduce pero' una base minima, sufficiente a:

- aggiungere `source_claim_id` e colonne analoghe
- creare `citation_checks`
- evitare che i DB gia' esistenti restino in stato misto v6/v7

**Safety contract**

- tutte le migration di uno step logico devono stare dentro una transazione esplicita
- `schema_version` viene aggiornato solo dopo `COMMIT`
- in caso di errore la migration deve fare rollback completo dello step corrente
- il runner deve essere sicuro anche se setup viene rieseguito dopo un'interruzione

**File owner**

- `plugin/scripts/setup.js`
- `plugin/db/schema.sql`
- eventuale nuovo helper `plugin/lib/migrations.js`

**Acceptance criteria**

- un DB v6 gia' esistente puo' essere portato a v7 senza reset manuale
- le nuove colonne necessarie risultano effettivamente presenti dopo setup
- il runner e' idempotente e non rompe reinstall/update successivi

### WP-00B — Structured Block Foundation

**Goal**  
Congelare il contratto del blocco strutturato e fornire il parser condiviso prima che partano i WP di ingestion.

**Perche' esiste**  
Claim, seed e review condividono lo stesso paradigma di artefatto embedded. Se il parser comune non ha un owner esplicito, i WP divergono.

**Output richiesti**

1. modulo condiviso `plugin/lib/structured-block-parser.js`
2. parser per `vibe-claim`, `vibe-seed`, `vibe-review`
3. contratto minimo degli errori di parsing e delle normalizzazioni comuni
4. esempi canonici riusabili nei test e nella skill
5. policy esplicita di tolleranza per varianti LLM ragionevoli

**Decisione di spec**

Questo WP non implementa claim ingestion, seed ingestion o review ingestion.

Implementa solo la fondazione comune che i tre WP importeranno.

Policy di tolleranza:

- `vibe-claim`, `vibe-seed`, `vibe-review` sono i tag canonici
- alias semplici come `claim`, `seed`, `review` possono essere accettati con warning nel solo v7
- campi extra vengono ignorati se non rompono il parse
- campi opzionali mancanti producono output normalizzato con `null`
- YAML malformato non rompe il hook: produce warning e delega al fallback compatibile del WP chiamante
- un blocco valido ma nel file sbagliato non deve essere ingestito dal modulo sbagliato

**File owner**

- `plugin/lib/structured-block-parser.js`
- eventuale documentazione in `evals/harness/`

**Acceptance criteria**

- esiste un parser condiviso unico per tutti i blocchi embedded v7
- WP-01, WP-02 e WP-03 non devono implementare parser indipendenti concorrenti
- il contratto del parser e' congelato prima dell'avvio del lavoro parallelo sui tre ingestion module
- esistono test espliciti sulle varianti LLM piu' probabili e sugli errori di formattazione

### WP-01 — Claim Event Ingestion

**Goal**  
Popolare automaticamente `claim_events` durante l'uso normale del plugin.

**Perche' esiste**  
Senza `claim_events` vivi, tutto il resto del lifecycle e' zoppo.

**Output richiesti**

1. Nuovo modulo `plugin/lib/claim-ingestion.js`
2. Funzione canonica `extractClaimEventsFromToolWrite(...)`
3. Uso di `logClaimEvent(...)` in `db.js`
4. Integrazione nel flusso `post-tool-use.js`
5. Dedupe strategy esplicita
6. Aggiornamento della skill per produrre blocchi claim machine-readable embedded nel markdown
7. Normalizzazione canonica dei claim ID a formato `C-NNN`
8. Istruzione esplicita inserita nella sezione `CRYSTALLIZE` di `SKILL.md`, subito dopo le istruzioni su `CLAIM-LEDGER.md`

**Trigger runtime**

- `Write` o `Edit` su `CLAIM-LEDGER.md`
- opzionalmente altri artefatti claim-structured dichiarati dalla spec

**Data minima da estrarre**

- `claim_id`
- `event_type`
- `session_id`
- `confidence` se disponibile
- `r2_verdict` se disponibile
- `kill_reason` se disponibile
- `narrative`
- `timestamp`

**Regola importante**

La path preferita non e' il parsing di prosa libera.

v7 definisce che una claim transition valida deve avere almeno una forma parseabile.

Per il core v7, la forma canonica e':

- blocco strutturato embedded nel markdown umano

Il sidecar esterno resta opzionale e non e' il path primario di adozione.

Quindi una claim transition valida deve avere almeno:

- blocco strutturato embedded
- oppure, in compatibilita', parsing euristico del markdown libero

Il parser euristico su markdown libero e' consentito come fallback compatibilita', ma la skill va aggiornata per far si' che non diventi il path principale di v7.

**Compatibilita' di transizione**

Per tutta la linea v7:

- il parser strutturato e' preferito
- il formato legacy markdown libero resta supportato
- il fallback euristico non viene rimosso in v7
- i Golden Claims esistenti non devono smettere di essere eseguibili solo per il cambio di formato

**Conseguenza comportamentale intenzionale**

- riflessa in un messaggio di blocco piu' chiaro nello stop hook

### WP-05 — Citation Verification Engine + Gate L0

**Goal**  
Trasformare la citation verification da protocollo skill-level a enforcement-grade service del plugin.

**Perche' esiste**  
Questo e' uno dei gap piu' netti tra metodologia dichiarata e runtime reale.

**Output richiesti**

1. Nuovo modulo `plugin/lib/citation-engine.js`
2. Chain di verifica minima:
   - DOI -> `doi.org`
   - fallback -> CrossRef
   - PMID -> PubMed E-utilities
   - arXiv -> arXiv metadata endpoint
3. retraction flagging minimo
4. integrazione di `citation_checks` con `gate-engine.js`
5. nuovo check L0 runtime reale
6. test spike preliminare che verifichi che un hook Claude Code possa eseguire `fetch()` outbound entro i limiti di budget definiti

**Decisione di design**

La verifica non deve bloccare il tool per attese di rete lunghe in modo indiscriminato, ma non puo' nemmeno lasciare il gate in uno stato ambiguo.

Strategia:

- estrazione e insert `PENDING` nel hook
- tentativo di verifica **sincrono e bounded** sugli identificatori leggeri al momento del hook
- retry/backfill async per il bulk o per i casi rimasti irrisolti
- il gate non deve dipendere da una race condition non dichiarata

**Policy esplicita v7**

1. DOI, PMID e arXiv rilevati dentro lo stesso evento provano prima una verifica sync con budget di tempo stretto.
2. Se la verifica rapida riesce, la citazione passa subito a `VERIFIED` o `UNRESOLVED`.
3. Se la verifica rapida non conclude entro budget, la citazione resta `PENDING` e viene demandata al path async di retry.
4. `L0` blocca su citazioni `UNRESOLVED` o `RETRACTED`, ma non inventa blocker se una claim non ha ancora citazioni tracciate.
5. `D1` / promozione di claim blocca su qualunque citazione ancora non `VERIFIED`, inclusi i `PENDING`.
6. `D1` blocca anche quando una claim non ha alcuna citazione tracciata: la promozione richiede almeno una fonte verificabile.

**Contract di implementazione hook HTTP**

- v7 usa `fetch()` built-in di Node 18+ con `AbortController`
- timeout sync canonico: `3000ms` per richiesta
- budget massimo sync per evento hook: `5000ms`
- massimo `3` citazioni tentate in sync per singolo evento
- budget target totale del path `post-tool-use.js`: `<= 6000ms`
- nessun supporto proxy-specific garantito in v7.0
- timeout o errore di rete nel path sync non marcano subito `UNRESOLVED`: lasciano `PENDING` e demandano retry async
- il limite viene documentato esplicitamente nella documentazione tecnica

**Contingency plan**

WP-05 e' il WP con il rischio storico di slip piu' alto nel progetto.

Se il test spike mostra che l'HTTP outbound nei hook non e' abbastanza affidabile o compatibile con l'ambiente reale:

- WP-04 va comunque in produzione
- le citazioni restano persistite e tracciate come `PENDING`
- `D1` continua a bloccare claim con citazioni non `VERIFIED`
- la verifica passa temporaneamente a workflow manuale o semi-manuale
- il completamento pieno di WP-05 slitta fuori dal core `7.0.0` senza bloccare il resto del sistema circolatorio

**Gate policy**

- claim senza citazioni puo' restare in `CLAIM-LEDGER`, ma non puo' essere promosso
- claim con citazioni `UNRESOLVED` non puo' passare `L0`
- claim con citazioni `PENDING` non puo' passare `D1`
- claim con citazioni `UNRESOLVED` non puo' passare `D1`
- claim con citazioni `RETRACTED` deve produrre alert forte

**File owner**

- nuovo `plugin/lib/citation-engine.js`
- `plugin/db/schema.sql`
- `plugin/lib/gate-engine.js`
- `plugin/scripts/post-tool-use.js`
- `skills/vibe/assets/schemas/source-validity.schema.json` come riferimento, non come punto da rompere

**Acceptance criteria**

- esiste uno stato persistente di source validity
- `L0` non e' piu' solo documentale
- citazioni non verificate non passano inosservate nel runtime

### WP-08 — Retrieval Closure

**Goal**  
Rendere la memoria di richiamo finalmente utile, veloce e robusta nel runtime reale, senza dipendere solo dal path vettoriale fragile.

**Priority**  
P1, non foundation blocker di v7 core.

**Perche' esiste**  
Oggi la memoria e' piu' lessicale che semantica nella path reale e il path vettoriale dipende da una chain infrastrutturale fragile.

**Chiarezza di capability**

FTS5 non e' semantic retrieval.

In v7, `Tier 0` significa:

- retrieval lessicale molto migliore di `LIKE`
- ranking BM25 utile
- infrastruttura reale e robusta

Ma non significa:

- sinonimi veri
- conceptual match affidabile
- sostituzione del tier vettoriale

La distanza tra `Tier 0` e `Tier 1` e' **funzionale**, non solo prestazionale.

**Output richiesti**

1. base FTS5 reale in SQLite per full-text retrieval locale, senza dipendenze nuove
2. query path primario che usa FTS5/BM25 quando il supporto e' disponibile
3. perimetro di indexing esplicito e non rumoroso:
   - `sessions.narrative_summary` come fonte primaria
   - sottoinsieme high-signal di `spine_entries`, non tutta la tabella
4. per i `spine_entries`, il sottoinsieme iniziale canonico e' limitato a:
   - `BUG_FIX`
   - `DESIGN_CHANGE`
   - `REVIEW`
   - `FINDING`
   - `GATE_CHECK` solo quando `gate_result` e' `WARN` o `FAIL`
5. schema FTS5 v7 con metadata minimi direttamente nel virtual table tramite colonne `UNINDEXED`; in v7 non si introduce come obbligo una tabella companion separata se `rowid` + metadata unindexed bastano
6. cap esplicito del testo indicizzato: massimo `2000` caratteri per entry, con truncation marker
7. scelta tokenizer documentata e testata su fixture scientifica minima; il default candidato e' `porter unicode61`, ma la decisione finale deve tenere conto di termini composti tipici del dominio (`CRISPR-Cas9`, `scRNA-seq`, `IL-6`, `p-value`)
8. creazione reale di `vec_memories` quando disponibile
9. query path che usa davvero query embedding come tier opzionale superiore
10. fallback utile su `memory_embeddings` e, in ultima istanza, su LIKE legacy
11. test su piattaforme supportate

**Decisione di spec**

Questa capability entra in v7 solo se non destabilizza il core e non blocca I1/I3/I4.

La fondazione pragmatica di v7 non e' `sqlite-vec`.

La fondazione pragmatica di v7 e':

1. **Tier 0** — FTS5 in SQLite, sempre quando supportato dal runtime reale
2. **Tier 1** — vector retrieval opzionale quando `sqlite-vec` + embedding pipeline sono realmente disponibili
3. **Tier 2** — fallback legacy su path lessicali minimi

Questo riduce la fragilita' infrastrutturale di WP-08 e permette un miglioramento reale anche senza chiudere subito tutto il path vettoriale.

La ragione principale per cui `Tier 0` vale la pena non e' la pura velocita' su corpus piccoli.

La ragione principale e':

- ranking utile
- query piu' pulita
- fondazione robusta per corpus che cresceranno

`WP-08` non viene giudicato riuscito se FTS5 diventa solo un `LIKE` piu' veloce ma rumoroso.

La lettura del repo IronMem conferma che FTS5 e' abbastanza forte da fungere da retrieval base in un sistema reale. Vibe Science prende da li' la lezione infrastrutturale, ma non adotta il loro pattern di iniezione via `IRONMEM.md`, la compressione LLM-first a ogni sessione o la semantica fail-silent dei hook.

**File owner**

- `plugin/db/schema.sql`
- `plugin/scripts/setup.js`
- `plugin/scripts/worker-embed.js`
- `plugin/lib/vec-search.js`
- `plugin/scripts/prompt-submit.js`
- `plugin/lib/context-builder.js`

**Acceptance criteria**

- e' documentato esplicitamente che `Tier 0` offre keyword-ranked retrieval, non semantic equivalence
- l'indice FTS5 non e' costruito su tutto indiscriminatamente: include `narrative_summary` e solo spine entries ad alto segnale
- la struttura usa `rowid` + metadata `UNINDEXED` in modo sufficiente per il v7 baseline, senza introdurre complessita' superflua
- il tokenizer scelto non rompe in modo evidente il retrieval su una fixture minima di termini scientifici composti
- ogni entry indicizzata rispetta il cap di lunghezza
- retrieval FTS5 reale e queryabile nel runtime di riferimento
- recall vettoriale reale quando l'ambiente lo supporta
- fallback non morto quando non lo supporta

v7.0 puo' dirsi completato solo se tutte queste condizioni sono vere:

1. il lifecycle `claim/review/seed` e' vivo nel DB
2. `L0` esiste davvero nel plugin per la validita' delle fonti
3. i Golden Claims non sono piu' solo un linter YAML
4. esiste una baseline v6.0 e una comparazione v7
5. il fail-open invisibile e' stato ridotto almeno a fail-loud osservabile
6. i DB v6 esistenti possono migrare senza reset manuale
7. nessuna nuova capability rompe l'identita' del progetto

---

## 13. Criteri di esclusione

Una implementazione NON entra in v7 se:

- aggiunge complessita' senza chiudere un gap reale
- richiede un nuovo cambio di identita' del progetto
- migliora packaging ma non runtime
- crea una superficie di manutenzione sproporzionata
- non ha criterio di verifica

---

## 14. Decisioni esplicite prese da questa spec

Per evitare future ambiguita', questa spec decide esplicitamente:

1. **I1, I3, I4 sono il cuore di v7.**
2. **I2 non e' foundation blocker.**
3. **I5 non entra nel core di v7.**
4. **Strict mode parte minimo, non come matrice tripla.**
5. **Il benchmark attuale non basta e va riclassificato.**
6. **Per v7 core, il formato strutturato canonico e' embedded nel markdown, non sidecar-first.**
7. **Il plugin deve preferire artefatti strutturati alla prosa libera.**
8. **Il readiness gate v7 esiste come comando unico, non come checklist dispersa.**
9. **Il bump stabile a `7.0.0` avviene solo dopo readiness verde e acceptance globale soddisfatta.**
10. **WP-05 ha un contingency path: WP-04 puo' andare live anche se la verifica hook-based resta temporaneamente incompleta.**
11. **WP-08 usa FTS5 come fondazione pragmatica; il vector path e' un tier superiore, non l'unica speranza.**
12. **Il README e la documentazione devono riflettere il runtime reale, ma la spec non e' subordinata al README.**

---

## 15. Conclusione

v7 non deve essere la versione con piu' feature.

v7 deve essere la versione in cui Vibe Science smette di avere i suoi gap piu' seri proprio nei punti che definiscono la sua identita':

- claim lifecycle
- citation rigor
- benchmark truth
- integrity runtime visibility

Se v6 ha costruito il corpo del plugin, v7 deve costruire il suo **sistema circolatorio**: far scorrere davvero claim, review, seed, source validity e benchmark dentro una architettura unica, verificabile e degna di un tool da campioni.
